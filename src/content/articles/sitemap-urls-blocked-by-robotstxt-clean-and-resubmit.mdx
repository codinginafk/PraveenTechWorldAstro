## Direct Answer

Have you ever found yourself staring at Google [Search Console](/articles/google-search-console-not-showing-data-8-fixes), only to see that your sitemap URLs are blocked by robots.txt? That can be a real head-scratcher! But don't worry--I've got some tips to help you clean up your sitemap and resubmit it. The main idea is to remove any URLs from your sitemap that aren't meant to be there, especially those that are blocked, duplicate, private, or non-canonical. After that, you might need to tweak your robots.txt file, but only if it's blocking a useful page by mistake. Then, you can test the affected URLs in Google [Search Console](/articles/how-to-enable-google-search-console-for-your-website) and resubmit your sitemap. Simple enough, right?
## Explanation

So, what's going on here? An XML sitemap is essentially a wish list for Googlebot--it's a list of URLs you'd like Google to find. On the flip side, robots.txt is a set of rules that tells Googlebot what it can and cannot crawl on your site. If there's a match between a Disallow rule in robots.txt and a URL in your sitemap, Googlebot can't crawl that URL, even though it's listed in the sitemap. 

According to [Google Search Central](https://developers.google.com/search/docs/crawling-indexing/robots/intro), these robots.txt rules dictate what Googlebot can crawl. 

This kind of issue can pop up after you've made some changes to your CMS, migrated your site, or updated an SEO plugin. Suddenly, your sitemap might start including unwanted pages like tag pages, author pages, or even old URLs that should be long gone. 

For instance, if your sitemap includes `/tag/seo-tips/`, but your robots.txt has this rule:

```text
Disallow: /tag/
```

Googlebot will be unable to crawl that URL. The fix? Just remove `/tag/seo-tips/` from your sitemap. 

Another example is if you have a live page under `/blog/`, but robots.txt has:

```text
Disallow: /blog/
```

In this case, your sitemap might be fine, but robots.txt is too broad. You would want to edit robots.txt to allow Google to crawl your actual blog pages. 

If you're still setting things up, it's a good idea to start with the basics.

## When This Fix Works

This approach works well when your sitemap includes pages you genuinely want Google to crawl, but robots.txt is mistakenly blocking them. For example, if you have a real service page at `/services/web-design/`, but robots.txt says `Disallow: /services/`, that's a problem you can fix! 

It's also helpful if your sitemap has junk URLs--things that should never be there, like tag pages or staging URLs. After a site migration, if your old sitemap still lists `/2022/old-post/` while robots.txt blocks `/2022/`, cleaning up the sitemap can resolve that conflict. 

The best outcome occurs when you address both sides: removing unwanted sitemap URLs and correcting overly broad robots.txt rules.

## When This Does NOT Work

Now, there are some situations where this fix won't be applicable. If a page is private and should remain that way--like an admin page or internal file--you'll want to keep the robots.txt rule and just remove it from the sitemap. 

Similarly, if a page has a noindex tag and you want it indexed, you'll need to remove that noindex tag first. If the server returns errors like 403, 401, or 500, that's a different issue altogether--it's a server or permissions problem, not a sitemap conflict. 

And if the canonical URL points elsewhere, Google might ignore your submitted URL in favor of the canonical version. For issues like that, you might want to explore further.

Lastly, if your sitemap file itself is broken--returning 404, 500, or invalid XML--Google [Search Console](/articles/how-to-fix-sitemap-errors-in-google-search-console) won't be able to process it correctly.

## Step-by-Step Instructions

Alright, let's break this down into some easy-to-follow steps:

1. **Open Google Search Console and select your property.**  
   Head over to [Google Search Console](https://search.google.com/search-console/). Make sure you pick the exact property for your site. If you have both `https://example.com` and `https://www.example.com`, choose the one matching your live site.

2. **Go to the Sitemaps report.**  
   In the left sidebar, click on **Sitemaps** under **Indexing**. Check your submitted sitemap status. If it says "Success" but still shows blocked URLs, head over to the **Pages** report next.

3. **Export the blocked URLs from GSC.**  
   Click on **Pages** in the left sidebar. Look for the section called **Why pages aren't indexed**. You might see entries like "Blocked by robots.txt." Click the row, and use the export button in the top right corner of the table. Starting with the first 50 URLs is usually enough to spot patterns without overwhelming you.

4. **Inspect a few blocked URLs.**  
   Go back to the left sidebar and click **URL Inspection**. Paste one of the blocked URLs into the search box at the top. Hit **TEST LIVE URL**. If it says the page is blocked by robots.txt, you've got a crawl rule conflict.

5. **Open your robots.txt file.**  
   Open a new tab and go to `https://yourdomain.com/robots.txt`, replacing `yourdomain.com` with your actual domain. If this file returns 404, Google treats it as if there are no rules, so the issue might be with your sitemap.

6. **Match the blocked URL against the Disallow rules.**  
   Use `Ctrl+F` on Windows or `Cmd+F` on Mac to search within robots.txt. If the blocked URL is `/blog/example-page/`, look for rules like `/blog/`, `/example-page/`, or any wildcard rules such as `/*`. 

7. **Decide which side is wrong.**  
   Ask yourself: should Google crawl this URL or not? If yes, you'll need to edit robots.txt. If no, remove it from the sitemap. Just be careful not to allow everything just to clear the warning!

8. **Fix robots.txt only for useful pages.**  
   If a real page is blocked, edit the rule in your hosting file manager, CMS editor, or security plugin. For WordPress users, here's a guide that might help: [link]. For example, change this:

```text
Disallow: /blog/
```

to this:

```text
Allow: /blog/
```

Or simply remove the `Disallow: /blog/` line if it's not needed.

9. **Remove bad URLs from the sitemap.**  
   Ensure your sitemap comprises indexable, canonical, public URLs. Get rid of tag pages, search pages, staging URLs, author archives, noindex pages, and old URLs that redirect. If you're using an SEO plugin, modify the sitemap settings there.

10. **Regenerate the sitemap.**  
    After adjusting the sitemap settings, regenerate it. Many plugins do this automatically. If you're using a static sitemap file, generate a fresh one and upload it to the same location, like `/sitemap.xml`.

11. **Check the sitemap in your browser.**  
    Open `https://yourdomain.com/sitemap.xml` to verify that it's working as expected.
