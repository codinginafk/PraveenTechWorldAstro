I want to share a little adventure I had automating TLS certificate renewal with a [Python script](/articles/automate-weekly-student-grade-reports-with-a-python-script-and-deepseek-prompts) generated by DeepSeek. It was a wild ride, and I'll admit it almost caused a production outage. Let me tell you what happened!

## Why I needed automation

Every 90 days, I would spend my Sunday afternoons manually renewing TLS certificates across three Nginx servers. You can probably guess how that went:
1. SSH into each server
2. Backup the current configs
3. Run `certbot renew`
4. Restart services
5. Repeat for each domain

It was such a tedious process, and I was always worried I might miss an expiration date. My monitoring system would alert me if any certs expired, but I wanted something that would just handle it all automatically at 2 AM, so I wouldn't be jolted awake at 3 AM.

## Getting DeepSeek to write the first draft

So, I decided to give DeepSeek a shot. I fed it a detailed prompt because, you know, the more context you give, the better the output:

```python
I need a Python script that:
1. Takes a list of domains (via CLI or config file)
2. Checks current cert expiration using openssl or certbot
3. Renews certs expiring within 30 days
4. Validates new certs
5. Gracefully restarts Nginx
6. Logs everything with timestamps
7. Has a dry-run mode
8. Uses argparse for CLI options
9. Handles errors gracefully
10. Provides a summary of successes/failures
```

The initial response was about 284 tokens long, which seemed promising. It even had a proper argparse setup and logging. I saved the script as `tls_renewal.py` and tested it locally. It worked fine in about 12 seconds, and the dry-run mode produced sensible output. Everything seemed ready for production!

## When the script failed spectacularly

I set up a cron job to run weekly at 2 AM:
```
0 2 * * 0 /usr/bin/python3 /opt/scripts/tls_renewal.py  - domains=example.com,api.example.com
```

But then, at 3 AM, I received a panicked call. The script failed immediately with an `ImportError: No module named 'cryptography'`. Digging deeper, I found a few critical issues:

1. The `check_expiry` function was completely fake - it always returned "45 days from now," so renewals never triggered.
2. Missing Python dependencies (`cryptography`) on production servers.
3. Inconsistent Nginx service names across servers (`nginx` vs `nginx.service`).
4. Permission issues with log files.

It was clear that the AI had provided me with a convincing prototype that crumbled under real-world conditions.

## How I fixed it

After realizing what had gone wrong, I spent about four hours transforming the prototype into something production-ready with three key improvements:

### 1. Real dependency management
I added a `requirements.txt` and automatic dependency checking:
```python
REQUIRED_LIBS = ["cryptography"]

def ensure_dependencies():
    missing = []
    for lib in REQUIRED_LIBS:
        try:
            __import__(lib)
        except ImportError:
            missing.append(lib)
    if missing:
        logging.critical(f"Missing libraries: {missing}")
        sys.exit(1)
```

### 2. Actual certificate checking
I replaced the placeholder logic with real certbot/openssl calls:
```python
def get_cert_expiry(domain):
    try:
        # Try certbot first
        result = subprocess.run(["certbot", "certificates", " - domains", domain], ...)
        # Parse actual expiry date from output
        return parse_date(result.stdout)
    except Exception:
        # Fall back to openssl
        result = subprocess.run(["openssl", "x509", "-noout", "-enddate", ...], ...)
        return parse_openssl_date(result.stdout)
```

### 3. Robust safety features
I added:
- Proper error handling
- Configurable Nginx service names
- JSON output for monitoring
- Better logging
- Dry-run improvements

## The final product

My production-ready script now:
1. Checks for required dependencies upfront
2. Uses real certificate expiration checks
3. Handles different Nginx service names
4. Provides detailed logging
5. Outputs machine-readable summaries
6. Has a comprehensive dry-run mode

To run it, you can do:
```bash
# One-time test
python3 tls_renewal.py  - domains=example.com  - verbose

# Weekly cron job
0 2 * * 0 /usr/bin/python3 /opt/scripts/tls_renewal.py >> /var/log/cron_tls_renewal.log 2>&1
```

## Key lessons learned

1. **AI needs very specific instructions** - My initial prompt didn't clearly specify real certificate checking, so the AI just provided a placeholder.
2. **Production is different from development** - Missing dependencies and environment differences can really break things.
3. **Safety features are mandatory** - The dry-run mode and good logging saved me hours of debugging.
4. **Monitoring is still needed** - I set up alerts to catch any failures in the automated process.
5. **Cost-effective** - The AI prompt cost me $0.04 and dependencies $0.02 - much cheaper than my time spent on manual renewals!

The AI gave me about 80% of what I needed, but that last 20% required my judgment and real-world experience. I'd love to hear your thoughts! Have you tried something like this?
