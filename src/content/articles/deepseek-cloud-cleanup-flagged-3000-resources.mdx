I recently tasked [DeepSeek](/articles/automated-tls-certificate-renewal-with-deepseek) with crafting a [Python script](/articles/automate-weekly-student-grade-reports-with-a-python-script-and-deepseek-prompts) to help me find orphaned cloud resources in an effort to trim our AWS bill. You would think this would be a straightforward task, right? Well, the first version of the script flagged an astonishing 3,000 resources for deletion  -  way off from the estimated 20 true orphans. Let's dive into what happened, and how I managed to fix it.

## What Problem Made Me Build This?
You know that creeping feeling when your AWS bill keeps climbing each month? That's exactly what I was dealing with. I had a hunch that orphaned resources, like forgotten EC2 instances, detached EBS volumes, and outdated snapshots, were hiding out there, accumulating costs like dust bunnies under a couch.

While exploring AWS Config and Trusted Advisor, I found both tools could identify unused resources. However, they often return a ton of false positives. In my case, I needed something that would truly understand our specific tagging conventions and how our teams operate.

Honestly, manually sifting through 11 AWS accounts across two regions would have taken ages. I was looking for a neat little script that could:

1. Scan all 11 accounts and both regions.
2. Identify EC2 instances, EBS volumes, load balancers, and snapshots with no recent activity.
3. Cross-reference against our tagging system to pinpoint actual owners.
4. Generate a clear report of what could be safely deleted.

So, I thought, why not let DeepSeek handle the heavy lifting? Sure, I can whip up basic boto3 calls, but navigating complex multi-account auth and API calls was a bit much for me.

## How I Prompted It

I crafted a detailed prompt to help DeepSeek understand what I was aiming for. Here's what I said:

```text
Prompt:
Write a Python CLI tool that finds orphaned cloud resources across multiple AWS accounts. Call it cleanup.py. It should:

1. Read a list of AWS account IDs and roles to assume from a config file (config.yaml).
2. For each account, assume the specified IAM role and scan us-east-1 and eu-west-1.
3. Scan the following resource types:
   a. EC2 instances (status, launch time, tags, last network activity)
   b. EBS volumes (attachment state, size, tags, last write time)
   c. ELB load balancers (number of targets, last activity, tags)
   d. EBS snapshots (age, volume status, tags, owning account)
4. For each resource, check if it has been idle for more than 30 days:
   a. EC2: stopped state OR no network bytes in 30 days
   b. EBS: unattached OR attachment count = 0
   c. ELB: zero registered targets OR no requests in 30 days
   d. Snapshot: older than 90 days AND source volume no longer exists
5. Flag any resource missing the 'Project' or 'Owner' tag.
6. Output a CSV report with columns: account, region, resource_type, resource_id, age_days, status, tags, reason, deletable (yes/no).
7. Include a  - dry-run flag that prints the report without any deletions.
8. Include a  - delete flag that actually terminates/deletes/deregisters the flagged resources.
9. Log everything to cleanup.log with timestamps.
10. Use boto3, PyYAML, and csv. Use only standard library beyond those.

Please output the complete script with comments and usage examples.
```

After running the script with ` - dry-run`, I was met with a staggering 900 lines of code. But that's where things started to get a bit dicey.

## Where Did the AI Output Break?
The script had some major hiccups that could've led to serious issues. Let's break it down:

### Hallucinated Tag Matching Logic

The AI assumed that any untagged resources would show up as `None`, which is a classic misunderstanding of the AWS API. In reality, if a resource has no tags, the `tags` field just shows up as an empty list. The script reported zero untagged resources, which in hindsight should have raised a red flag. I was too focused on the output numbers to notice the inconsistency.

To make matters worse, the AI added a filter that checked for `'Project' in tags` and `'Owner' in tags`, flagging anything that was missing either one. However, our organization has a different tagging structure. Some teams use `'CostCenter'` instead of `'Project'`, while others prefer `'Team'` over `'Owner'`. So, while the AI's logic was functioning, it simply didn't align with our environment.

### False Positive Cascade

Since the tag matching didn't work, the script couldn't properly determine the ownership of numerous resources, defaulting to "deletable" for anything it couldn't classify. This resulted in 3,012 flagged resources  -  a number that left me reeling. 

Upon inspecting the CSV, I found that it had flagged:

- **12 running EC2 instances** actively serving traffic but missing the 'Project' tag.
- **4 production load balancers** that seemed idle during maintenance windows.
- **2,800 EBS snapshots** that were over 90 days old  -  many from decommissioned accounts, but around 30 were still in active use.

Among the flagged resources, only about 200 were actual orphans. Finding them amongst the sea of false positives? Nearly impossible.

### No Pagination (Again)

The script only scanned the first page of results for each API call. AWS APIs paginate results, and for accounts with hundreds of resources, the script was only considering about 20% of the real inventory.

### Rate Limiting from Cross-Account Assumption

The script sequentially assumed IAM roles for each account without any backoff. When you switch roles too quickly, the AWS API throttles you, and boom  -  the script crashed on the fourth account with a `ThrottlingException`. No partial results, no retry.

## What I Had to Fix

I spent several evenings redesigning the script to tackle these issues.

**Tag matching** (2 hours): I replaced the AI's binary check with a configurable tag mapping. Now, the script reads a `tag_map.yaml` that defines which tags correspond to ownership for each account. 

**Idle detection** (1 hour): I improved the logic to check CloudWatch metrics for actual network traffic instead of just the instance state. A stopped instance behaves differently from a running-but-idle instance, and I introduced a configurable idle threshold.

**Snapshot filtering** (1.5 hours): I added a check against AWS Backup to see if each snapshot is part of a backup plan, which cut false positives from 2,800 down to just 40.

**Pagination** (1 hour): I implemented pagination loops using AWS's built-in paginators, ensuring that each resource type was properly handled.

**Cross-account rate limiting** (30 minutes): I added a random delay between 2-5 seconds between account switches to avoid throttling.

**Output** (30 minutes): The CSV output was enhanced to group resources by account and include a confidence score for each deletion recommendation.

After all these changes, the final script spanned about 1,600 lines across three files:

- `cleanup.py`: main CLI and orchestration
- `scanners.py`: resource-specific scanning logic
- `report.py`: CSV report generation and formatting

## The Actual Impact

After the overhaul, the script successfully identified 23 genuine orphaned resources:

- 7 unattached EBS volumes (costing about $35/month)
- 3 stopped EC2 instances with attached EBS volumes (costing around $120/month)
- 2 old load balancers with zero targets (about $45/month)
- 11 orphaned EBS snapshots (costing roughly $200/month)

While the $400/month savings were a win, the ultimate success was establishing a regular cleanup process across teams. Now, I can run this script monthly to catch orphans before they add up.

I even added a Slack integration that summarizes the potential deletions each Monday, giving the team 48 hours to voice any concerns before the cleanup actually occurs.

## What I Learned

The AI made some bold assumptions that didn't hold up. It misinterpreted tag names, applied a one-size-fits-all idle threshold, and misunderstood pagination across AWS services.

The dry-run flag was invaluable. Seriously, if I had run the first version without ` - dry-run`, we would have lost 12 active EC2 instances. The AI was ready to delete resources that were actively serving traffic, simply because they were missing the specific tags it was programmed to look for.

The ongoing lesson I take from my DeepSeek experiments is that while AI can churn out infrastructure code quickly, it doesn't grasp the intricacies of your specific environment. Each organization has its quirks - custom tagging, varying backup policies, and teams that might not always follow the guidelines. The AI can't know these unless you spoon-feed it every little detail, and even then, it might still get things wrong.
