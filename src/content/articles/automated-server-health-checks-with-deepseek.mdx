I recently decided to automate my server health checks using DeepSeek, and let me tell you, it was quite the journey! The first version of the script had some major hiccups that would have caused serious problems under real server loads. After some debugging and adjustments, I now have a lightweight health checker that monitors four servers and only alerts me when there's an actual issue. 

## What Made Me Want to Do This?
I manage a handful of Ubuntu servers for various projects. One of them is dedicated to monitoring, another houses a small e-commerce site, and the other two are for internal tools for clients. Every Saturday morning, I found myself SSHing into each server and running the same commands:

```
df -h
free -m
systemctl status --failed
uptime
```

This manual process was a bit tedious and took about 15 minutes. I kept telling myself that I should set up Nagios or Zabbix, but honestly, that felt like overkill for just four servers. I wanted something simpler--a single [Python script](/articles/automate-weekly-student-grade-reports-with-a-python-script-and-deepseek-prompts) that I could run from my laptop to check all servers, gather the metrics, and alert me if anything was off.

## The Prompt I Gave to DeepSeek

I opened up DeepSeek and crafted a single prompt that captured everything I wanted:

```
Prompt:
Build a Python CLI tool that connects to multiple Linux servers via SSH and runs health checks. The tool should:

1. Accept a YAML or JSON config file listing servers (host, port, user, key_path).
2. For each server, collect:
   - CPU load (load average from /proc/loadavg)
   - Disk usage (df -h output, parse percentage)
   - Memory usage (free -m output, parse used/total)
   - Running services (systemctl list-units --state=running, count)
   - Failed services (systemctl --failed, list them)
3. Compare each metric against configurable thresholds (cpu_load > 2.0, disk_usage > 80%, memory_usage > 90%).
4. If any threshold is exceeded, send a Slack webhook with the server name, metric, current value, and threshold.
5. Run in a loop with a configurable interval (--interval default 300 seconds).
6. Log all checks to a file named health-check.log with timestamps.
7. Use async SSH connections (asyncssh) so all servers are checked in parallel, not sequentially.
8. Include a --once flag to run a single check and exit.
9. Include a --version flag.

Please output the complete script with comments and a usage example.
```

DeepSeek generated this impressive 450-line script with all the features I asked for, including async SSH connections and threshold checks. It even came with a colorful summary table! The output cost me around $0.002 on DeepSeek's API, which isn't too shabby.

## What Went Wrong Initially

Unfortunately, the first version had a few critical issues:

1. **SSH Connection Pooling:** While the AI used asyncssh, it created a fresh connection for every single metric check. On the first run, my laptop opened 12 simultaneous SSH connections to each server, which caused my client's server to flag it as a brute force attempt and block my IP. I received an error message that read:
   ```
   Connection refused by remote host. Possible rate limiting.
   ```

2. **Threshold Logic Was Inverted:** The script used OR logic for triggering alerts, which meant if one metric was high (like CPU), it would alert even if all the others were fine. This led to me getting bombarded with 47 identical Slack messages within 10 minutes!

3. **No Error Handling on SSH Failures:** When one server was down for maintenance, the script crashed entirely instead of skipping it and checking the others. The unhandled exception caused the whole loop to stop.

   ```
   Traceback (most recent call last):
     File "health_checker.py", line 203, in check_server
       async with asyncssh.connect(host, port=port, username=user, client_keys=[key_path]) as conn:
   asyncssh.misc.PermissionDenied: Permission denied (publickey).
   ```

## My Fixes

After realizing all these issues, I made four key improvements:

1. **Connection Reuse:** I rewrote the SSH logic to open one connection per server, run all the checks through that single connection, and then close it. This reduced the number of connections from 12 to just 3 per cycle.

2. **Alert Deduplication:** I added a mechanism to track sent alerts by server and metric. If an alert had already been sent within the last hour, it would skip it. This change reduced my Slack notifications from 47 to just 1.

3. **Graceful Failure Handling:** I wrapped each server check in a try/except block to log any failures and continue with the others, so one down server wouldn't derail the entire process.

4. **Parallel Execution:** I retained DeepSeek's async structure but switched from asyncio.gather() to asyncio.wait() with return_when=FIRST_EXCEPTION, allowing me to collect results even when some servers failed.

The final script is now 4.1KB and checks four servers in about 6 seconds (a significant improvement from the original 45 seconds). Plus, memory usage remains under 40MB!

## What I Learned About Crafting Prompts for Health Check Scripts

1. **Be Clear About Connection Management:** I learned that DeepSeek assumed every metric needed its own SSH connection, so grouping them together in a single statement helped a lot.

2. **Deduplication Needs to Be Explicit:** I had to clarify in the prompt that I wanted to track which alerts had been sent to avoid duplicates.

3. **Consider Unreachable Servers:** The AI didn't initially account for what happens when a server is down, so adding that to the prompt was crucial.

4. **Cost vs. Time:** While DeepSeek's output cost me $0.002, it took me 90 minutes to debug and fix the issues. Writing everything from scratch would have taken about 3 hours, so I saved some time, but there were trade-offs.

## FAQ

**Q: Does this script work on Windows servers'**  
A: Not directly. The script uses /proc and subprocess calls that assume a Linux environment. For Windows, you would need to swap the disk check to wmic or PowerShell Get-CimInstance and change the service check to Get-Service.

**Q: Can I run this without email or Slack configured'**  
A: Yes. Omit the --webhook flag and the script logs everything to health.log and stdout. You can pipe the output to any other tool.

**Q: How much CPU does this use'**  
A: On a 2-core VPS, the script uses about 0.8% CPU during each check cycle and 35MB of RAM. The sleep interval between checks keeps it idle 99% of the time.

**Q: What happens if a service restarts while the script is checking'**  
A: The script takes a snapshot of service states at the moment of the check. If the service was down for 2 seconds during restart, it reports it as down. I added a 3-second cooldown to avoid alerting on legit restarts.

**Q: Is there a Docker version'**  
A: Not yet, but you can wrap it in a Docker container with a simple COPY of the script and a CMD entrypoint. The script only needs Python 3.8+ and the requests library for webhooks.
