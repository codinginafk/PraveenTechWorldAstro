I recently set out on a journey to build a Python script that tracks our AI API costs across four different providers using DeepSeek. After a week of late-night debugging (thanks to some interesting AI 'hallucinations'), I managed to save my team about $2,000 per month in unnecessary spending. Curious about how it all unfolded? Let me share my experience!

## The Problem That Started It All

So, here's the deal: our team was burning through AI API credits faster than I could keep track. We had engineers using OpenAI for various tasks, a research project leveraging Anthropic Claude, my own experiments with DeepSeek, and the data team diving into Google's Gemini. The end result? Financial chaos!

Invoices were arriving in different formats at totally unpredictable times, and when I finally added everything up, I was shocked—$4,700 per month! Almost half of that went to premium models for tasks that really didn’t need them.

What I needed was a command-line tool that could:
1. Pull usage data from all four providers
2. Normalize costs into a single view
3. Identify our most expensive models
4. Generate a simple HTML dashboard for the team

This felt like a job for DeepSeek! While I only have basic Python skills, I thought having AI handle the tedious work of wiring together four different API SDKs with authentication and cost normalization was a smart move.

## Building the Tracker with DeepSeek

I decided to craft a detailed prompt for DeepSeek, hoping to generate a solid foundation to build upon:

```text
Prompt:
Write a Python CLI tool that tracks AI API costs across multiple providers. Call it track.py. It should:

1. Support OpenAI, Anthropic (Claude), Google (Gemini), and DeepSeek APIs.
2. Read API keys from environment variables (OPENAI_API_KEY, ANTHROPIC_API_KEY, etc.)
3. Accept a --days flag (default 7) to look back N days.
4. For each provider:
   a. Authenticate and call their usage/list endpoints
   b. Extract: model name, prompt tokens, completion tokens, total cost
   c. Handle 401 (bad key - skip provider), 429 (rate limit - retry with backoff), 5xx (log and continue)
5. Normalize all costs to USD.
6. Print a summary table grouped by provider showing:
   - Total tokens (input/output split)
   - Total cost
   - Cost per million tokens
7. Include a --serve flag that generates a standalone HTML dashboard file (dashboard.html) with charts showing daily cost trends.
8. Log all API errors to cost-tracker.log with timestamps.
9. Use only standard library plus requests, with fallback handling if requests is missing.
10. Output colorized status messages.

Please output the complete script with comments, a sample config, and usage examples.
```

To my surprise, DeepSeek generated about 800 lines of code using standard libraries like `requests` and `datetime`. It even included a cute animated spinner for data fetching. I thought, "This is too good to be true!" 

## Where the AI Code Fell Apart

But when I ran `python track.py --days 30` with my actual API keys, the issues began to surface.

### Imaginary API Endpoints

First, I discovered that DeepSeek had completely fabricated OpenAI's usage API endpoint:
```
openai/usage?date=2026-06-01
```
This URL doesn't exist! OpenAI's usage data actually comes from their billing dashboard API or CSV exports. The AI generated something that looked plausible but was purely fictional.

### Cost Calculation Errors

Then, I noticed that the script assumed all providers used the same pricing formulas. In reality, it was way more complicated:
- Anthropic rounds token counts per request
- Google returns costs in micros (millionths of a cent)
- DeepSeek provides token counts but no cost data

### Pagination Problems

When I tested it with real production data, it only captured the most recent 100 records from each provider. Each API handles pagination differently, which the AI code overlooked completely:
- OpenAI uses cursor-based pagination
- Anthropic uses offsets
- Google uses page tokens
- DeepSeek uses limit/offset

### Rate Limit Chaos

With 12 API keys across four providers, the script's basic `time.sleep(1)` approach didn’t cut it. We quickly hit:
```
429 Too Many Requests
```
The single retry attempt after 5 seconds was insufficient for our actual usage patterns.

## The Fixes That Made It Work

After six evenings of debugging, here’s what I changed:

**API Endpoints (2 hours)**: I replaced all fictional endpoints with the real ones from each provider's documentation.

**Cost Calculations (1 hour)**: I built provider-specific cost calculators with hardcoded pricing tables (as of June 2026) and added a `--refresh-prices` flag to keep them updated.

**Pagination (2 hours)**: I implemented proper pagination for each provider using generator patterns to handle their different approaches.

**Rate Limiting (1 hour)**: I swapped the basic sleep for a proper token-bucket system that respects each provider's limits and rotates through available keys.

**Dashboard (30 minutes)**: I rewired the AI-generated HTML template to use our actual normalized cost data instead of the hallucinated numbers.

The final version consisted of three files:
- `track.py` (600 lines) - Main CLI
- `providers.py` (500 lines) - Provider-specific logic
- `dashboard.py` (300 lines) - HTML generation

Testing against 90 days of real usage showed costs within 3-5% of provider dashboards.

## The Real Savings

With visibility into our costs, the waste became painfully clear:
- **OpenAI GPT-5**: A forgotten background job using GPT-5 instead of GPT-4-mini - $800/month
- **Anthropic Claude Opus**: Research team using Opus when Sonnet would suffice - $600/month
- **DeepSeek**: Double-paying through a 30% markup aggregator - $400/month
- **Google Gemini**: Unlimited test accounts running expensive experiments - $200/month

Total savings: around $2,000/month!

## Lessons Learned

This project taught me some valuable lessons about working with AI-generated code:

1. **AI can invent plausible-looking APIs** - That two-hour wild goose chase for a non-existent endpoint was a real eye-opener. The AI can be confidently wrong.

2. **Pricing changes frequently** - Hardcoded cost tables can go stale fast. The `--refresh-prices` flag became essential.

3. **Rate limiting can get complex quickly** - What works for one key may fail spectacularly with multiple. A proper token bucket system solved the issue.

4. **Pagination is a silent killer** - Without proper handling, you'll miss much of your data and never realize it.

5. **Visibility enables savings** - The real value came not just from tracking costs but in finding optimization opportunities, like that forgotten GPT-5 job.
