I built a Python script to track our AI API costs across four providers using DeepSeek. After a week of late-night debugging to fix the AI's hallucinations, it ended up saving my team about $2,000 per month in wasted spending. Here's how it all went down.

## The Problem That Started It All

Our team was burning through AI API credits at an alarming rate. We had engineers using OpenAI for various tasks, a research project running on Anthropic Claude, my own experiments with DeepSeek, and the data team testing Google's Gemini. The result? Complete financial chaos.

Each provider sent invoices in different formats at random times, with nobody keeping track. When I finally tallied everything up, the number made me physically recoil - $4,700 per month, with nearly half going to premium models for tasks that didn't need them.

I needed a CLI tool that could:
1. Pull usage data from all four providers
2. Normalize costs into a single view
3. Identify our most expensive models
4. Generate a simple HTML dashboard for the team

This was perfect for DeepSeek. While I know basic Python, wiring together four different API SDKs with authentication, rate limiting, and cost normalization sounded like exactly the kind of tedious work I wanted AI to handle.

## Building the Tracker with DeepSeek

I crafted a detailed prompt outlining the entire architecture, hoping to get a working skeleton I could build upon:

```text
Prompt:
Write a Python CLI tool that tracks AI API costs across multiple providers. Call it track.py. It should:

1. Support OpenAI, Anthropic (Claude), Google (Gemini), and DeepSeek APIs.
2. Read API keys from environment variables (OPENAI_API_KEY, ANTHROPIC_API_KEY, etc.)
3. Accept a --days flag (default 7) to look back N days.
4. For each provider:
   a. Authenticate and call their usage/list endpoints
   b. Extract: model name, prompt tokens, completion tokens, total cost
   c. Handle 401 (bad key - skip provider), 429 (rate limit - retry with backoff), 5xx (log and continue)
5. Normalize all costs to USD.
6. Print a summary table grouped by provider showing:
   - Total tokens (input/output split)
   - Total cost
   - Cost per million tokens
7. Include a --serve flag that generates a standalone HTML dashboard file (dashboard.html) with charts showing daily cost trends.
8. Log all API errors to cost-tracker.log with timestamps.
9. Use only standard library plus requests, with fallback handling if requests is missing.
10. Output colorized status messages.

Please output the complete script with comments, a sample config, and usage examples.
```

DeepSeek returned about 800 lines of seemingly perfect code using standard libraries like `requests` and `datetime`. It even included a cute animated spinner for data fetching. Too good to be true? Absolutely.

## Where the AI Code Fell Apart

The moment I ran `python track.py --days 30` with my actual API keys, the problems started piling up.

### Imaginary API Endpoints

First crash: DeepSeek had completely fabricated OpenAI's usage API endpoint:
```
openai/usage?date=2026-06-01
```
This URL doesn't exist. OpenAI's actual usage data comes through their billing dashboard API or CSV exports. The AI generated something that looked plausible but was pure fiction.

### Cost Calculation Errors

The script assumed all providers used identical pricing formulas. Reality was much messier:
- Anthropic rounds token counts per request
- Google returns costs in micros (millionths of a cent)
- DeepSeek gives token counts but no cost data

### Pagination Problems

When testing with real production data, it only captured the most recent 100 records from each provider. Turns out every API handles pagination differently:
- OpenAI uses cursor-based pagination
- Anthropic uses offsets
- Google uses page tokens
- DeepSeek uses limit/offset

The AI code ignored all these variations.

### Rate Limit Chaos

With 12 API keys across four providers, the script's naive `time.sleep(1)` approach failed spectacularly. We immediately hit:
```
429 Too Many Requests
```
The single retry attempt after 5 seconds couldn't handle our actual usage patterns.

## The Fixes That Made It Work

Six evenings of debugging later, here's what I changed:

**API Endpoints (2 hours)**: Replaced all fictional endpoints with real ones from each provider's documentation.

**Cost Calculations (1 hour)**: Built provider-specific cost calculators with hardcoded pricing tables (as of June 2026) and added a `--refresh-prices` flag to update them.

**Pagination (2 hours)**: Implemented proper pagination for each provider using generator patterns to handle their different approaches.

**Rate Limiting (1 hour)**: Swapped the basic sleep for a proper token-bucket system that respects each provider's limits and rotates through available keys.

**Dashboard (30 minutes)**: Rewired the AI-generated HTML template to use our actual normalized cost data instead of hallucinated numbers.

The final version spanned three files:
- `track.py` (600 lines) - Main CLI
- `providers.py` (500 lines) - Provider-specific logic
- `dashboard.py` (300 lines) - HTML generation

Testing against 90 days of real usage showed costs within 3-5% of provider dashboards.

## The Real Savings

Once we had visibility, the waste became obvious:
- **OpenAI GPT-5**: A forgotten background job using GPT-5 instead of GPT-4-mini - $800/month
- **Anthropic Claude Opus**: Research team using Opus when Sonnet would suffice - $600/month
- **DeepSeek**: Double-paying through a 30% markup aggregator - $400/month
- **Google Gemini**: Unlimited test accounts running expensive experiments - $200/month

Total savings: ~$2,000/month.

## Lessons Learned

This project taught me some hard truths about working with AI-generated code:

1. **AI invents plausible-looking APIs** - That two-hour wild goose chase for a non-existent endpoint won't be my last. The AI is confidently wrong.

2. **Pricing changes constantly** - Hardcoded cost tables go stale fast. The `--refresh-prices` flag became essential.

3. **Rate limiting gets complex fast** - What works for one key fails spectacularly with twelve. Proper token buckets per key solved it.

4. **Pagination is a silent killer** - Without proper handling, you'll miss most of your data and never know.

5. **Visibility enables savings** - The real value wasn't tracking costs, but finding optimization opportunities like that forgotten GPT-5 job.
