# The 14KB Website: My Journey from Bloat to Barebones HTML

Let's be honest - modern websites have gotten ridiculously heavy. The average page now weighs over 2MB, and my personal site was even worse. I had built it as a React single-page app with 87 npm dependencies. The Webpack build took 14 seconds, and Lighthouse gave me a depressing 38/100 on mobile. I was spending more time updating packages than writing content.

So I nuked it. I rebuilt everything with just four plain HTML files, added SQLite via WebAssembly for search functionality, and hosted it on the edge. The result? A featherlight 14KB site that loads before you can blink. No frameworks, no bundlers, no node_modules nightmares.

Here's why this matters.

## The Numbers Don't Lie

| Metric | Old React Site | New Vanilla Site |
|--------|----------------|------------------|
| Page weight | 2,341 KB | 14 KB |
| DOM ready | 4.8s | 0.3s |
| Lighthouse score | 38 | 99 |
| Build time | 14s | None |
| Dependencies | 87 | 0 |
| Files to deploy | 1,142 | 4 |
| Hosting cost | $12/month | Free |

That 99.4% reduction changes everything. My old site needed a loading spinner. The new one shows content before your browser finishes processing the HTML.

## What Can You Fit in 14KB?

Fourteen kilobytes is smaller than most website icons. Smaller than Google's tracking scripts. Smaller than a single Tailwind CSS file. Here's what I managed to include:

- **index.html**: Main page with essential CSS baked in
- **blog.html**: Blog listing with pre-rendered summaries  
- **search.html**: Full-text search using SQLite/WASM
- **contact.html**: Basic email form

For styling? A 20-line CSS reset in a `<style>` tag, semantic HTML selectors, and two CSS variables for colors. The entire design fits in 187 lines of CSS and works flawlessly on any device.

## The SQLite WASM Trick

Here's the clever bit: instead of setting up a search backend, I use SQLite compiled to WebAssembly. Here's how search works:

1. Your browser downloads a 1.8MB `.wasm` file (cached after first load)
2. Queries run against a local 340KB SQLite database
3. Results appear in under 50ms - no waiting for a server

This is edge computing stripped to its essentials: your browser becomes the database server. Here's how it's implemented:

```html
<script type="module">
import sqlite3InitModule from 'https://esm.sh/@sqlite.org/sqlite-wasm@3.45.0/sqlite-wasm.js';
const sqlite3 = await sqlite3InitModule();
const db = new sqlite3.oo1.DB('/search-index.sqlite3');
const results = db.exec(`SELECT title, url FROM posts WHERE content MATCH ?`, { bind: [query] });
</script>
```

No servers. No ORMs. Just a file and a query.

## Why We're Stuck in Framework Land

We've accepted that websites need hundreds of KBs of JavaScript just to display text. The average React page loads 327KB before showing anything. Even "static" frameworks like Next.js come with hydration overhead.

Here's the uncomfortable reality: most content sites don't need frameworks. My new site handles:

- Dark mode toggle: 12 lines
- Mobile menu: 8 lines  
- Search functionality: 40 lines

Total: 60 lines of plain JavaScript. Zero dependencies. Zero build steps.

## Real Performance Gains

The difference was night and day:

- Core Web Vitals pass rate: 23% → 100%
- Largest Contentful Paint: 4.2s → 0.6s  
- Layout shift: 0.18 → 0.02

Testing on a budget Android phone over 3G:
- React site: 19 seconds to interactive
- Vanilla site: 1.8 seconds to interactive

## When This Approach Doesn't Work

This isn't a universal solution. If you need:
- Live updates
- Complex client-side state
- Collaborative features

...use a framework. But for blogs, documentation, and marketing sites? The framework overhead isn't worth your visitors' bandwidth.

## How to Build Your Own

1. **Start with HTML** - Just semantic tags, no templates
2. **Inline essential CSS** - 20-40 lines in a `<style>` tag
3. **Use one JS file** - Skip jQuery/Alpine for basic UI
4. **Let browsers handle routing** - Anchor tags work fine
5. **Host on the edge** - Cloudflare/Netlify/Vercel
6. **Measure everything** - Aim for under 100KB total

## Why This Matters

The web was designed for documents. My 14KB site:
- Loads in one network request
- Works without JavaScript
- Passes all accessibility checks

The React version actually failed accessibility because screen readers couldn't wait for hydration to finish.

Bloat isn't accidental - it's a choice. Every dependency makes your visitors pay for your convenience. For content sites, that trade-off rarely makes sense.

## Common Questions

**Q: Is 14KB realistic with images/fonts?**  
A: No - this covers just HTML/CSS/JS. Use system fonts and optimized images. The goal isn't hitting 14KB exactly, but questioning every unnecessary byte.

**Q: What about SEO?**  
A: Google loves fast, semantic HTML. My indexed pages increased 40% because crawlers didn't need to execute JavaScript.

**Q: Handling comments?**  
A: Use GitHub issues, a lightweight API, or pre-generate comments with SQLite.

**Q: Is SQLite WASM secure?**  
A: Yes - it runs in the browser sandbox with no server access.

**Q: Works for e-commerce?**  
A: Not for checkout flows, but perfect for product/marketing pages.
