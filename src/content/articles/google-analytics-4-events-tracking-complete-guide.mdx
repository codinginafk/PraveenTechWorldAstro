## GA4 Events Are Different From Everything You Knew

If you used Universal Analytics, you are used to thinking in page views, sessions, and bounce rates. GA4 throws that model out completely. Everything in GA4 is an event. Page views are events. Clicks are events. Scrolls are events. Purchases are events.

This shift is powerful once you understand it. Instead of tracking pages, you track what people actually do. But the terminology and setup process confuse most people at first. This guide covers exactly how events work in GA4 and how to set them up correctly.

## How GA4 Events Work

In GA4, an event is any interaction that your tracking code sends to Google. Each event has a name and can include parameters that provide additional information.

A page view in GA4 is an event called `page_view`. It includes parameters like `page_title`, `page_location`, and `page_referrer`. A purchase event called `purchase` includes parameters like `value`, `currency`, and `transaction_id`.

GA4 groups events into four categories. Automatically collected events fire without any configuration. Enhanced measurement events come from GA4's built-in tracking toggles. Recommended events are predefined names that unlock specific GA4 features. Custom events are anything you define yourself.

Understanding this hierarchy is essential because Google uses it to determine which events get special treatment in reports. If you use a custom event name for a purchase when the recommended event name `purchase` exists, you lose access to built-in ecommerce reporting.

For troubleshooting when events are not showing up, see our [GA4 troubleshooting guide](/blog/ga4-not-tracking-visitors-12-troubleshooting-steps).

## Automatically Collected Events You Get for Free

GA4 collects several events automatically as soon as you add your tracking code. You do not need to configure anything for these.

`first_visit` fires the first time a user visits your site. `session_start` fires when a new session begins. `page_view` fires when a user loads a page. `user_engagement` fires when a user has the page in focus for at least one second.

These events give you basic traffic data without any setup. You can view them in the Realtime report and in standard exploration reports. They are the foundation everything else builds on.

## Enhanced Measurement Events

Enhanced measurement is GA4's easiest way to add meaningful event tracking without writing code. You enable it in your data stream settings under the Enhanced Measurement section.

Each enhanced measurement option tracks a specific type of interaction. Page views tracks URL changes for single-page applications. Scrolls fires a `scroll` event when a user reaches 90 percent page depth. Outbound clicks tracks clicks on links that go to other domains.

Site search tracks when a user uses your internal search feature. Video engagement tracks YouTube video plays, pauses, and completions embedded on your site. File downloads tracks clicks on common file type links.

Enable all of these unless you have a specific reason not to. Each one adds valuable behavioral data with zero code. The only catch is that enhanced measurement uses browser history events for SPA page tracking, which may not catch every route change in complex single-page apps.

## Recommended Events and When to Use Them

Google provides recommended event names for common business actions. Using these names correctly gives you automatic access to GA4 reporting features and Google Ads integration.

For ecommerce, the recommended events include `view_item`, `add_to_cart`, `remove_from_cart`, `begin_checkout`, `add_shipping_info`, `add_payment_info`, and `purchase`. Each has specific required and recommended parameters that GA4 uses to populate its ecommerce reports.

For lead generation, use `generate_lead` when a user submits a form. Include the `value` and `currency` parameters if you can assign a monetary value to the lead.

For content sites, track `login`, `sign_up`, `share`, and `select_content` events. These tell GA4 how users engage with your content beyond just reading it.

The full list of recommended events with their parameters is in Google's documentation. Match your event names and parameters exactly. GA4 does not recognize `addtocart` as the same as `add_to_cart`.

## Custom Events and Event Parameters

Custom events are everything else you want to track that is not covered by automatic, enhanced, or recommended events. You create custom events by sending them from your site code or through Google Tag Manager with an event name you define.

To send a custom event with gtag.js, use `gtag('event', 'your_event_name', {parameter: 'value'})`. With Google Tag Manager, create a new GA4 Event tag with your event name and parameters.

Event parameters are key-value pairs that add context to events. For a custom event called `button_click`, you might include parameters like `button_text`, `button_location`, and `page_section`.

Parameters become available in reports only after you register them as custom dimensions or metrics. Without registration, parameters exist in the event data but do not appear in standard reports. You can still access unregistered parameters in BigQuery exports if you have that integration enabled.

GA4 limits custom event parameters to 100 per event and 25 text parameters per property for reporting. Plan your parameter strategy to stay within these limits while capturing the data that matters most.

## How to Verify Your Events Are Working

Never trust that your events are working because you set them up correctly. Always verify with GA4's built-in debugging tools.

DebugView is your primary verification tool. Enable debug mode on your site by adding `?gtm_debug=x` if using GTM or installing the Google Analytics Debugger Chrome extension. Open GA4 DebugView and perform the action you want to track. If the event appears, your tracking is working.

For more thorough testing, use the Realtime report. It shows the last 30 minutes of events. Click around your site and watch for events appearing in the event count by event name card.

For implementation issues, use Google Tag Manager Preview mode. It shows which tags fire on each page and what data they pass. If your GA4 event tag fires but the event does not appear in DebugView, the problem is in how the tag sends data, not in the tag firing.

## Event Tracking Best Practices

Plan your events before you implement them. Create a tracking plan document that lists every event name, its parameters, and what action triggers it. This prevents the chaos of random event names and inconsistent parameters.

Use consistent naming conventions. All lowercase with underscores. Descriptive but not verbose. `purchase` is better than `purchase_completed_successfully`.

Do not create events for everything. More events mean more data to sift through and more potential points of failure. Track what you will actually use to make decisions.

Mark key events as conversions. In GA4, go to Configure and then Conversions. Mark the events that represent completed business goals. This enables them in reporting and makes them available for bid optimization in Google Ads.

Document your implementation. Keep a record of what you track, where the code lives, and when you set it up. Six months from now, you will not remember why you created that custom event. Your documentation saves future you hours of confusion.

## FAQ

**Q: What is the difference between a recommended event and a custom event in GA4?**
**A:** Recommended events are predefined event names Google suggests for common actions like login, purchase, or add_to_cart. They come with recognized parameters that unlock GA4 features automatically. Custom events are any event you define yourself with your own event name and parameters. Recommended events get better integration with Google's reporting and advertising features.

**Q: How many custom events can I create in GA4?**
**A:** GA4 allows up to 500 unique event names per property. If you need more, you must remove unused events or upgrade to GA4 360. There is no limit on the total number of event instances, only on the number of distinct event names.

**Q: Do I need to track every single interaction on my site?**
**A:** No. Track only events that align with your business goals. A content site should track article reads, scroll depth, and newsletter signups. An ecommerce site needs product views, add to cart, and purchases. Tracking everything creates noise that makes it harder to find meaningful data.

**Q: Why are my custom events not showing up in GA4 reports?**
**A:** Custom events do not appear in standard reports until you mark them as conversions or create custom exploration reports. GA4 only automatically surfaces a few default events like page_view, session_start, and first_visit in standard reports. For everything else, you need to configure reporting manually.

**Q: Can I edit or delete events after they are created in GA4?**
**A:** You cannot delete events from the GA4 event list, but you can modify event parameters and create event modifications in the admin panel. To stop an event from being tracked, remove the code that sends it from your site or GTM configuration.
