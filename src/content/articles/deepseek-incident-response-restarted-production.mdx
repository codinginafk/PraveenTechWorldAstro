## Why I wanted an automated incident response script

If you've ever been jolted awake at 3 AM by a server issue, you know the drill. A service goes down, the monitoring tool goes wild, and I find myself SSHing in, checking logs, restarting services, and then trying to catch some more sleep. But every morning, I realize the process is the same. So, I thought, why not automate it?
I manage about 12 Linux servers with a mix of PostgreSQL, Nginx, Redis, and a custom Python API behind Gunicorn. When things go south, I typically need to:

1. Confirm if the service is truly down or just sluggish.
2. Rotate the logs to prevent disk space issues.
3. Restart the service.
4. Send a summary to our team Slack channel.

I figured a single CLI tool could tackle all four tasks. So, I turned to DeepSeek, thinking it would whip up a working script in no time. Spoiler alert: I was wrong.

## The exact prompt I gave DeepSeek

After my morning coffee, I felt pretty optimistic as I crafted my prompt:

```text
Prompt:
Write a Python CLI tool called 'incident_responder.py' that automates common incident response tasks on Linux servers.

The script should:

1. Accept a service name as an argument (e.g., postgresql, nginx, redis).
2. Check the current status of the service using systemctl.
3. If the service is inactive or failed:
   a. Rotate the service's log files (move the current log to a .old file).
   b. Restart the service using systemctl restart.
   c. Verify the service started successfully.
   d. Send a Slack notification with the service name, action taken, and timestamp.
4. If the service is already running, print "OK" and exit.
5. Use argparse with  - service as a required argument and optional  - slack-webhook for the Slack URL.
6. Keep it under 40 lines.
7. Use subprocess.run for all system commands.
8. Handle common errors like service not found or permission denied.

Please provide the complete script with comments.
```

The AI churned out the code in about 11 seconds. It looked neat, and I was excited to test it out. I quickly ran: `python3 incident_responder.py  - service postgresql`. And then... things got a bit wild.

## Where the AI made dangerous mistakes

### Mistake 1: It restarted production during peak hours

So, the script ran at 2:15 PM on a Tuesday. Our PostgreSQL serves an e-commerce API that sees around 400 requests per second during peak hours. The AI's script checked the service status and said "OK."

I then intentionally stopped PostgreSQL to see how the restart logic would work. The AI's script immediately executed `systemctl restart postgresql`. No checks for maintenance windows, no confirmations, nothing. Just a restart command. 

I had set ` - slack-webhook` to a test channel, so I received a message that read: "Restarted postgresql at 2026-06-28T14:15:32Z". But there was no context - it didn't mention dropped connections. My colleague quickly pinged me: "Did you just restart the DB?"

That would have disrupted around 400 active connections. Although our application has connection pooling and retry logic, it would still lead to latency issues at a critical time. Not ideal when customers are trying to check out.

### Mistake 2: It rotated logs that were 2 hours old

Next up was the log rotation logic, which was another shocker. The AI's script did this:

```python
def rotate_logs(service):
    log_path = f"/var/log/{service}/{service}.log"
    if os.path.exists(log_path):
        os.rename(log_path, f"{log_path}.{int(time.time())}")
        print(f"Rotated {log_path}")
```

It renamed the active log file, including all the crucial crash logs from the last 2 hours. The very logs I would need for debugging were just gone. Linux has `logrotate` for a reason, and the AI completely missed that point.

### Mistake 3: No health check before declaring success

After restarting the service, the AI's script checked `systemctl is-active postgresql` and deemed it a success if the output was "active." But here's the kicker: PostgreSQL can report as "active" while still refusing connections. The AI didn't verify whether the service was fully operational, which is pretty critical.

### Mistake 4: It silently skipped permission errors

The AI included try-except blocks, but they were quite generic:

```python
except Exception as e:
    print(f"Error: {e}")
```

This meant that errors were only printed and not logged or sent to Slack. If there was an issue with the Slack webhook (which, spoiler alert, there was during my first test), the script just exited without a clue. If this had run from a cron job, I would have missed the error entirely.

## What I had to fix

I ended up spending about 3 hours fixing the AI's output. Here are the changes I made:

**1. Added a maintenance window check.**

I introduced two environment variables: `MAINTENANCE_WINDOW_START=02` and `MAINTENANCE_WINDOW_END=05`. The script now checks the current hour against these. If a restart is requested outside this window, it logs a CRITICAL alert and sends a Slack message, but doesn't restart. It waits for human intervention, which feels much safer.

**2. Replaced the log rotation with a proper copy-and-truncate.**

Instead of renaming the whole log file, the script now:

```python
def safe_rotate(service):
    log_path = f"/var/log/{service}/{service}.log"
    rotated_path = f"/var/log/{service}/{service}.log.old"
    try:
        subprocess.run(["cp", log_path, rotated_path], check=True)
        subprocess.run(["truncate", "-s", "0", log_path], check=True)
    except subprocess.CalledProcessError as e:
        log_error(f"Log rotation failed for {service}: {e}")
```

This keeps the crash logs intact while clearing the active file - much better for debugging!

**3. Added a real health check.**

After a restart, the script now attempts to connect to the service on its port. For PostgreSQL, it tries a `psql` connection. For Redis, it sends a `PING`. For Nginx, it checks the HTTP status code on the health endpoint. Only if that succeeds does it report the service as recovered.

**4. Replaced silent error handling with structured logging.**

Every error now gets logged to three places: a local error log file, the terminal output, and the Slack webhook if configured. Even if Slack fails, the error is recorded locally, which is a great safety net.

**5. Added a  - dry-run flag.**

The first thing the script does now is check for ` - dry-run`. If set, it prints what it WOULD do without executing any commands. This allows me to review the sequence of actions before committing to them, which feels way safer.

## What I learned about prompting AI for ops automation

1. **The AI does not understand production.** It treats every server like a dev environment. You have to explicitly teach it about maintenance windows, connection draining, and graceful degradation.

2. **Log rotation is simple until it's not.** The AI might delete your crash logs because "rotate" sounds like "move out of the way." Be very clear about what happens to each byte.

3. **Health checks must be real.** Just checking systemctl status isn't enough. The AI will consider the service healthy as long as the process is running. You need application-level health checks.

4. **Error handling is crucial.** The AI's try-except blocks were far too generic. For a cron job, errors need to be logged properly, or you might miss them entirely.

5. **Dry-run mode is essential.** Trusting an AI-written script with your production servers is risky. Always run the first test with ` - dry-run`, even after reviewing the code.

## FAQ

**Can I run the final script safely on my production servers'**  
Yes, but only after you configure the MAINTENANCE_WINDOW_START and MAINTENANCE_WINDOW_END variables. The script will refuse to restart any service outside those hours. I also recommend running it with  - dry-run for the first week to make sure the maintenance window covers your actual low-traffic period.

**What if a service is already down outside maintenance hours'**  
The script flags it as CRITICAL in the report but does not restart it automatically. I made that decision deliberately after the AI's first attempt. You get a ping on Slack and you decide whether to restart manually. That trade-off keeps us safe even if it means I sometimes get paged at 3 AM.

**Does this script work on Windows servers'**  
No, it is written for Linux systemd-based systems. The AI originally tried to use `sc.exe` commands which work on Windows but the service status parsing was completely different. If you need Windows support, you would need to adapt the service checking logic to use PowerShell's Get-Service cmdlet.

**How do I add a new service check'**  
Append the service name to the SERVICES array at the top of the script. The AI originally hardcoded each service in a separate function, which made maintenance painful. My version uses a configurable list so adding 'nginx' or 'postgresql-16' is one line.

**What happens if the Slack webhook URL is wrong'**  
The script catches the HTTP error and falls back to writing the alert to a local file called 'missed_alerts.log'. The AI did not handle this at all - it just crashed with an unhandled exception. That was one of the first things I fixed.
