When I moved from Universal Analytics to GA4, the first thing I noticed was that my page view numbers looked different. Not wrong, just different. It took me a while to understand why.

GA4 approaches page tracking differently than Universal Analytics. While UA treated page views as distinct hits, GA4 categorizes them as events—altering how you analyze the data.

## Page Views Are Now Events

In Universal Analytics, a page view was a special type of hit. You had a dedicated pageview tracking code, and it showed up in its own report section. In GA4, everything is an event. A page view is just a `page_view` event.

This shift is significant because GA4 events include parameters. For instance, loading a page triggers a `page_view` event containing details such as:
- `page_location` — the full URL
- `page_title` — the browser tab title
- `page_referrer` — where they came from

You can use these parameters in reports and custom analyses in ways that UA never allowed.

## Enhanced Measurement Covers Most Needs

GA4 includes enhanced measurement, which means you get page tracking without adding any code. If you have the basic GA4 tag installed, page views are tracked automatically.

To check if enhanced measurement is on:

1. Go to Admin > Data Streams in GA4
2. Click your web data stream
3. Look for "Enhanced measurement" — it should show as enabled
4. Inside, verify that "Page views" is checked

If you see zero page view data, start with the [GA4 troubleshooting guide](/blog/ga4-not-tracking-visitors-12-troubleshooting-steps) before digging into configuration.

## How Sessions and Page Views Work Together

This is where GA4 differs most from Universal Analytics.

In UA, a session was a strict 30-minute window. Every page view within that window belonged to the same session. If someone opened a page, left for 35 minutes, and came back, it was a new session.

GA4 uses a different model. A session ends after 30 minutes of inactivity, but page views are tracked independently. One session can include multiple page views. One page view can start a new session.

In practice, GA4’s page view totals will align roughly with UA’s, though session metrics might vary due to the updated tracking model. This is normal. Compare GA4 data to GA4 data over time, not to your old UA numbers.

## Page Tracking for Single Page Applications

If your site uses JavaScript to load content without full page reloads (like React, Vue, or other frameworks), GA4's automatic page tracking may not catch every "page view." This is because the browser does not send a new page load event — it just swaps content.

The fix is to send custom `page_view` events in your SPA router. Every time the URL changes, call:

```
gtag('event', 'page_view', {
  page_location: window.location.href,
  page_title: document.title
})
```

I had to do this on a client project built with Next.js. Without it, GA4 showed about 60% of actual page views.

## Common Page Tracking Problems

**Duplicate page views.** If you see double counts, check that you do not have the GA4 tag installed twice. This happens when you add it manually AND through Google Tag Manager, or when you have both a plugin and manual code.

**Missing page titles.** If your reports show "(not set)" for page titles, the `page_title` parameter is not being populated. Check that your page has a proper title tag and that the GA4 tag fires after the title is set.

**Wrong page paths.** If URLs look strange in reports, your site may be using hash-based routing or query parameters that confuse GA4. Use the `page_path` parameter override in your tracking code to clean up URLs.

## Frequently Asked Questions

**Q: Why does GA4 show fewer page views than Universal Analytics?**
A: GA4 filters out spam and bot traffic more aggressively than UA did. It also treats sessions differently. A 5-15% difference is normal. If the gap is larger, check for implementation issues like missing enhanced measurement or duplicate tags.

**Q: Can I track PDF downloads as page views in GA4?**
A: GA4 treats file downloads as separate events, not page views. Enhanced measurement automatically tracks downloads as `file_download` events. You can report on these separately or create a custom exploration that combines page views and downloads.

**Q: Does GA4 track page scrolls as part of page views?**
A: No. Scrolls are tracked as separate `scroll` events via enhanced measurement. A page view fires once when the page loads. Scroll tracking shows you how far people actually read. Both are useful but they measure different things.

**Q: How long does it take for page view data to appear in GA4 reports?**
A: Standard reports update within a few hours. Real-time reports show data instantly. If you set up tracking today, expect to see meaningful data in 24 to 48 hours. Processing delays are normal.

**Q: Can I use both UA and GA4 page tracking at the same time?**
A: Yes, and you should during the transition. They use separate tracking codes and do not interfere with each other. Running both side by side lets you compare data and build a history in GA4 before UA stops processing new data.

## Comparing GA4 Page Views Across Reports

One thing that confused me early on was seeing different page view numbers in different GA4 reports. The Realtime report shows one number, the Engagement report shows another, and the Pages and Screens report shows a third.

This is normal. Each report uses a different data processing level:

- **Realtime:** Shows events within the last 30 minutes. Data is not fully processed and may change.
- **Standard reports:** Show fully processed data with up to 24 hours of delay.
- **Explorations:** Let you build custom queries with raw event data.

The most accurate page view count for analysis is in the Engagement > Pages and Screens report after 24-48 hours of processing.

## Debugging Your Page Tracking Implementation

When page tracking breaks, I check these things in order:

1. **Is the GA4 tag actually loading?** Install the GA4 DebugView extension or use Chrome DevTools to verify the gtag script loads and fires events.

2. **Are events appearing in DebugView?** Open GA4 > Admin > DebugView. If you see events from your device, tracking is working. If not, the tag may not be installed correctly.

3. **Is enhanced measurement enabled?** Go to Admin > Data Streams and confirm Page Views is checked under Enhanced Measurement.

4. **Is there a filter excluding your traffic?** Internal traffic filters can accidentally exclude all data if misconfigured. Check Admin > Data Settings > Internal Traffic.

5. **Is consent mode blocking tracking?** If you use Consent Mode v2 and the user denies analytics cookies, GA4 may not fire. I added a cookie consent banner that only blocks tracking when explicitly declined.

## Page Tracking in the GA4 Data Model

Understanding how GA4 stores page view data helps you interpret reports correctly. When a `page_view` event fires, GA4 records:

- The event name (`page_view`)
- Event parameters (`page_location`, `page_title`, `page_referrer`)
- User identity (anonymous or logged-in)
- Session information
- Timestamp

This data flows into GA4's event-based schema. Unlike Universal Analytics, which stored page views in a separate table, GA4 treats everything as events in a single data stream.

You can query this data directly using GA4's Explorations or export it to BigQuery for advanced analysis. The BigQuery export includes raw event data with all parameters, which is useful for building custom reports that the GA4 interface does not support.
