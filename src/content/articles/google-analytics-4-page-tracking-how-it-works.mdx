## How Page Tracking Works in Google Analytics 4

Page tracking is the foundation of any analytics setup. Without accurate page view data, you cannot measure traffic, user behavior, or content performance. GA4 handles page tracking differently than Universal Analytics did, and understanding these differences is essential for setting up correct tracking on your website.

In GA4, every interaction is an event. A page view is a specific type of event called `page_view`. When a visitor loads a page on your site, the GA4 tracking code automatically fires this event along with parameters like the page URL, page title, and referrer information. These parameters allow you to analyze which pages perform best, where users enter and exit your site, and how they navigate between pages.

If your GA4 property is not showing any data, read our [GA4 troubleshooting guide](/blog/ga4-not-tracking-visitors-12-troubleshooting-steps) before proceeding.

## The Page View Event: What Gets Collected

When a `page_view` event fires, GA4 automatically captures several parameters:

- **page_location:** The full URL of the page being viewed.
- **page_title:** The title tag of the page.
- **page_referrer:** The URL of the page the user came from, if available.

These parameters appear in your GA4 reports under Reports > Engagement > Pages and Screens. You can see which pages get the most views, which have the highest engagement rate, and how users move between them.

In addition to the standard parameters, you can add custom parameters to page_view events. For example, you might pass a `content_type` parameter to distinguish blog posts from product pages or landing pages. This requires modifying your tracking code to include custom parameters in the config command.

## Enhanced Measurement and Automatic Page Tracking

GA4's enhanced measurement feature, enabled by default when you create a web data stream, automatically tracks several interactions that complement page views:

- **Page views:** The core `page_view` event fires on every page load.
- **Scrolls:** Fires when a user reaches 90% of the page depth.
- **Outbound clicks:** Tracks when a user clicks a link to an external domain.
- **Site search:** Captures search queries made on your internal search bar.
- **Video engagement:** Tracks YouTube video interactions.
- **File downloads:** Records downloads of documents and media files.

To verify enhanced measurement is enabled, go to Admin > Data Streams > select your web stream. Under Enhanced measurement, ensure the "Page views" toggle is on. This toggle controls the automatic `page_view` event that fires on every standard page load.

For a complete overview of GA4 setup and reporting, see our [Google Analytics for beginners guide](/blog/google-analytics-for-beginners-how-to-track-your-website-traffic).

## Session Management in GA4 Page Tracking

Sessions in GA4 are defined differently than in Universal Analytics. A session starts when a user first visits your site. The session continues until 30 minutes of inactivity pass, at which point a new session begins on the next interaction. You can adjust this timeout in Admin > Data Settings > Session timeout settings.

Page views play a key role in session management. Every `page_view` event resets the session timer. If a user navigates through 10 pages on your site within 30 minutes, that counts as one session with 10 page views. If the user leaves and returns after 31 minutes, the next page view starts a new session.

This matters for your analytics because session-centric metrics like "pages per session" and "engagement rate per session" depend on accurate session management. If your session timeout is too short, you inflate your session count. If it is too long, you undercount sessions.

## Single-Page Application (SPA) Page Tracking

SPAs present a challenge for traditional page tracking because they do not trigger a full page reload when the user navigates. Instead, the URL changes via the browser's History API.

GA4 handles SPA navigation through enhanced measurement. When enabled, the "Page changes based on browser history events" toggle listens for URL changes via the History API and fires a new `page_view` event each time the URL changes. This ensures that virtual page views are tracked even though no full page load occurs.

If you use a modern frontend framework like React, Vue, or Angular, check that this setting is enabled in your data stream settings. For complex SPA implementations with hash-based routing, you may need to manually push `page_view` events using the `gtag('config', 'G-XXXXXXXXXX', {page_path: '/new-path'})` command.

For more advanced tips on configuring GA4 for your specific site, check our [GA4 content strategy guide](/blog/how-to-use-google-analytics-4-to-improve-your-content-strategy).

## Debugging Page Tracking Issues

If your page view data looks incorrect, several common issues could be the cause.

**Missing page_view events:** Check that your GA4 tracking code is installed correctly and appears on every page of your site. Use the Realtime report in GA4 to verify that events are firing when you browse your site. If the Realtime report shows no activity, your tracking code is not installed or is blocked.

**Duplicate page_view events:** This happens when you have two copies of the tracking code on the same page, or you are using both gtag.js and Google Tag Manager simultaneously. Search your site source code for your measurement ID. If it appears more than once per page, remove the duplicate.

**Incorrect page_location values:** Some CMS platforms and SPA frameworks modify URLs in ways that confuse GA4. Check the page_location parameter in DebugView to see the actual URL being sent. If it includes query parameters or fragments that should be excluded, configure URL exclusions in your data stream settings.

**Missing enhanced measurement events:** If scrolls, outbound clicks, or file downloads are not appearing in your reports, re-check the enhanced measurement settings in your data stream. Some tracking options are toggled individually and may have been turned off during initial setup.

## How to Use Page Tracking Data to Improve Your Site

Once page tracking is working correctly, the data can guide your content and optimization decisions.

**Identify your best-performing pages.** Sort the Pages and Screens report by views. Your most-viewed pages show what topics resonate with your audience. Create more content in those areas.

**Find navigation bottlenecks.** Use the Path exploration report to see how users move between pages. If a high percentage of users leave from a specific page without visiting others, that page may have poor internal linking or a weak call to action.

**Optimize your landing pages.** The Landing Page report shows the first page users see when they arrive. If a landing page has a high page view count but low engagement rate, the page may not deliver on the promise of the search result or marketing link that brought users there.

**Track page performance over time.** Set your data retention to 14 months in Admin > Data Settings. This allows you to compare page view trends year-over-year and identify seasonal patterns in your content.

## Frequently Asked Questions

**Q: Does GA4 track page views automatically?**
**A:** Yes. When you install the GA4 tracking code (gtag.js) or set up a Google Tag Manager GA4 tag, the page_view event fires automatically on every page load. You do not need to write any extra code for basic page view tracking. Enhanced measurement adds automatic tracking for scrolls, outbound clicks, site search, file downloads, and video engagement without additional configuration.

**Q: How is GA4 page tracking different from Universal Analytics?**
**A:** Universal Analytics used a pageview hit type that was session-based. GA4 uses an event-based model where page_view is just one of many possible events. This means GA4 can track multiple interactions within a single session more accurately. GA4 also handles single-page applications (SPAs) better by supporting history change events that fire when the URL changes without a full page reload.

**Q: Why is GA4 showing fewer page views than my old Universal Analytics property?**
**A:** GA4 is stricter about what counts. It uses different session timeout settings (default 30 minutes of inactivity) and does not count sessions from bots and crawlers as accurately filtered out. Some privacy-focused browsers and ad blockers block GA4 more aggressively than they blocked Universal Analytics. Seeing 15-30% fewer page views in GA4 is normal.

**Q: How do I track page views on a single-page application (SPA) with GA4?**
**A:** GA4 handles SPAs better than Universal Analytics did. Enable enhanced measurement and make sure the 'Page changes based on browser history events' toggle is turned on in your data stream settings. This fires a page_view event whenever the URL changes via the History API. For complex SPAs, you may need to manually push page_view events using gtag config commands.

**Q: How can I verify that GA4 is tracking page views correctly on my site?**
**A:** Use the Realtime report in GA4. Open your site in a browser tab, then check GA4 > Reports > Realtime. If you see yourself listed as an active user, page_view events are firing. For detailed debugging, use the Google Tag Assistant Chrome extension or enable DebugView in GA4 by adding ?gtm_debug=x to your URL (if using GTM).

GA4 page_view event documentation is available from Google ([Google Analytics Help](https://support.google.com/analytics/answer/9191807)).
